Bienvenue sur le site web du projet GeoIdO

Nous entrons aujourd’hui dans l’internet du monde connecté : l’internet des objets (ou «Internet Of Things» : IoT).

L'IoT ?

L'Internet des Objets est constitué d'un réseau d'objets physiques (véhicules, machines, électroménager ou autres), qui utilisent des capteurs et des API pour se connecter et échanger des données via Internet.
Les appareils intelligents génèrent des volumes colossaux de données IdO à analyser et exploiter en temps réel avec les outils d'analyse prédictive et du Big Data.

Exploiter les données IoT ?

L'exploitation de cette masse de données nécèssite l'utilisation de serveurs afin d'en faciliter la gestion. L'OGC a donc mis en place des normes relatives au serveur "sensorthings" afin de mettre en place une unité dans le domaine.

Le projet GeoIdO

Objectifs

Le projet R&D GeoIdO est né du choix de Magellium d’être compétitif dans un domaine d’avenir, celui du monde des objets connectés créant de la donnée. L’entreprise doit pour cela acquérir plusieurs compétences technologiques :

• Maitriser l’utilisation des flux de données provenant des réseaux IoT
• Maitriser les API compatibles avec les serveurs IoT
• Créer un client IoT

Les grandes étapes en bref

• Installation d'un serveur FROST
• Implémentation du code permetant d'alimenter la BDD au travers du serveur FROST. Les données utilisées ont été récupérées sur site Atmo d'Auvergne-Rhone-Alpes (lien du site) au format .csv
• Implémentation du client permettant la visualisation de données.

Les choix

Serveur FROST plutôt de GOST

Les serveurs GOST et FROST sont tout deux composés d’une partie «sensing » permettant de récupérer et de traiter les données et « tasking » permettant de piloter les capteurs. Nous n’avons cependant eu accès qu’à la documentation de la partie sensing du serveur FROST qui nous a été fournit par l’institut Fraunhofer. Documentation : lien documentation.

Choix du langage pour la communication avec le serveur FROST

Nous avons choisit ensuite dans la documentation le langage JavaScript/jQuery car nous connaissions déjà ce langage et que nous manquions de temps. Cependant, il aurait été utile de comparer l’utilité du langage JavaScript/jQuery avec celle des langage cURL et HTTP afin de compléter notre choix.

Choix de la méthode d’installation du serveur

Nous avons opté pour la création de guide d’installation car nous n’avions pas d’autre idée au départ.

Les pistes d'améliorations

Opter pour un language différent pour la communication avec serveur FROST

Envisager un langage SQL (via un FME) qui serait plus approprié pour une grande quantité de données. Les requêtes SQL sont en effet plus rapides.

Utiliser une machine virtuelle pour installer le serveur FROST

la solution consistant à utiliser des VM et à transmettre une image de la VM serait beaucoup plus simple car elle ne dépendrait pas du système d’exploitation.

Exploiter de la donnée en temps réel

L’objectif (l’intérêt) de l’IoT est de faire du traitement de données en temps réel. Cependant, un tel traitement nécessite un accès aux flux de données IoT qui ne sont pas gratuits. De plus, il faut être capable de gérer la récupération de données sur un site en temps réel. Une façon de faire pourrait être de générer au travers d’un site des données d’un format équivalent à celui d’un flux de données.

Glossaire

API

En informatique, l’interface de programmation applicative (en anglais API) est un ensemble normalisé de classes, de méthodes ou de fonctions qui sert de façade par laquelle un logiciel offre des services à d’autres logiciels.

OGC (Open Geospatial Consortium)

L’OGC est une organisation internationale à but non lucratif engagée dans l’élaboration de normes ouvertes de qualité pour la communauté géospatiale mondiale. Ces normes sont élaborées par consensus et sont librement accessibles à quiconque pour améliorer le partage des données géospatiales du monde.
